# 充电站模拟器系统 - 需求文档

## 1. 项目概述

### 1.1 项目背景
充电站模拟器系统旨在提供一个可视化的充电站管理和充电过程模拟平台，帮助用户了解充电站的运行状态、充电过程以及车辆电池管理系统（BMS）的实时数据。

### 1.2 项目目标
- 实现充电站、充电桩、充电枪的层级管理
- 提供充电过程的可视化模拟和控制
- 实时展示充电数据和车辆 BMS 数据
- 支持多充电枪并发充电模拟

### 1.3 目标用户
- 充电站运营人员
- 充电设备测试人员
- 系统开发和维护人员

## 2. 功能需求

### 2.1 站点管理功能

#### 用户故事 2.1.1：查看充电站列表
**作为** 系统用户  
**我想要** 查看所有充电站的列表  
**以便于** 了解系统中有哪些充电站可用

**验收标准：**
- 系统应显示所有充电站的列表
- 每个充电站应显示：站点名称、地址、状态
- 列表应支持点击跳转到站点详情
- 站点数据应从数据库中查询获取

#### 用户故事 2.1.2：查看充电桩列表
**作为** 系统用户  
**我想要** 查看某个充电站下的所有充电桩  
**以便于** 了解该站点的充电桩配置情况

**验收标准：**
- 点击充电站后应显示该站点下的所有充电桩
- 每个充电桩应显示：充电桩编号、状态
- 充电桩列表应支持点击查看充电枪详情
- 应正确展示一对多的关系（一个站点对应多个充电桩）

#### 用户故事 2.1.3：查看充电枪列表
**作为** 系统用户  
**我想要** 查看某个充电桩下的所有充电枪  
**以便于** 选择可用的充电枪进行充电模拟

**验收标准：**
- 充电桩详情页应显示该充电桩下的所有充电枪
- 每个充电枪应显示：充电枪编号、类型（快充/慢充）、状态（空闲/充电中/故障）
- 充电枪应支持点击跳转到充电模拟器页面
- 应正确展示一对多的关系（一个充电桩对应多把充电枪）

### 2.2 充电模拟功能

#### 用户故事 2.2.1：插枪操作
**作为** 系统用户  
**我想要** 模拟将充电枪插入车辆的操作  
**以便于** 开始充电准备流程

**验收标准：**
- 充电模拟器页面应提供"插枪"按钮
- 点击插枪后应创建充电会话
- 系统应记录充电枪ID和车辆ID
- 插枪成功后应启用"启动充电"按钮
- 充电枪状态应从"空闲"变为"已插枪"

#### 用户故事 2.2.2：启动充电
**作为** 系统用户  
**我想要** 启动充电过程  
**以便于** 开始向车辆充电

**验收标准：**
- 插枪后应显示"启动充电"按钮
- 点击启动充电后应开始充电会话
- 系统应记录充电开始时间
- 充电枪状态应变为"充电中"
- 应建立 WebSocket 连接以接收实时数据
- 右侧车辆面板应从置灰状态变为正常显示

#### 用户故事 2.2.3：结束充电
**作为** 系统用户  
**我想要** 结束充电过程  
**以便于** 停止向车辆充电

**验收标准：**
- 充电中应显示"结束充电"按钮
- 点击结束充电后应停止充电会话
- 系统应记录充电结束时间和总电量
- 应断开 WebSocket 连接
- 应启用"拔枪"按钮

#### 用户故事 2.2.4：拔枪操作
**作为** 系统用户  
**我想要** 模拟拔出充电枪的操作  
**以便于** 完成整个充电流程

**验收标准：**
- 结束充电后应显示"拔枪"按钮
- 点击拔枪后应结束充电会话
- 充电枪状态应变回"空闲"
- 右侧车辆面板应变回置灰状态
- 页面应重置到初始状态

### 2.3 实时数据展示功能

#### 用户故事 2.3.1：展示充电实时数据
**作为** 系统用户  
**我想要** 实时查看充电过程中的电压、电流、功率等数据  
**以便于** 监控充电状态

**验收标准：**
- 启动充电后应在左侧面板显示充电数据
- 应显示以下数据：电压（V）、电流（A）、功率（kW）、SOC（%）
- 数据应通过 MQTT 消息实时更新
- 数据应通过 WebSocket 推送到前端
- 数据更新频率应不低于 1 秒/次
- 未充电时数据应显示为"--"

#### 用户故事 2.3.2：展示车辆 BMS 数据
**作为** 系统用户  
**我想要** 实时查看车辆电池管理系统的数据  
**以便于** 了解车辆电池状态

**验收标准：**
- 启动充电后应在右侧面板显示 BMS 数据
- 应显示以下数据：电池电压（V）、电池电流（A）、电池温度（°C）、SOC（%）
- 数据应通过 MQTT 消息实时更新
- 数据应通过 WebSocket 推送到前端
- 未充电时右侧面板应置灰显示
- 充电中右侧面板应正常显示

### 2.4 页面导航功能

#### 用户故事 2.4.1：站点列表导航
**作为** 系统用户  
**我想要** 从首页导航到站点列表页面  
**以便于** 开始使用系统

**验收标准：**
- 首页应提供到站点列表的入口
- 点击后应跳转到站点列表页面
- 页面应正确加载站点数据

#### 用户故事 2.4.2：充电桩导航
**作为** 系统用户  
**我想要** 从站点列表导航到充电桩列表  
**以便于** 查看具体的充电桩信息

**验收标准：**
- 点击站点后应跳转到充电桩列表页面
- 页面应显示该站点下的所有充电桩
- 应显示站点名称作为页面标题

#### 用户故事 2.4.3：充电模拟器导航
**作为** 系统用户  
**我想要** 从充电枪列表导航到充电模拟器页面  
**以便于** 开始充电模拟

**验收标准：**
- 点击充电枪后应跳转到充电模拟器页面

- 页面应初始化充电控制按钮状态

## 3. 非功能需求

### 3.1 性能需求

#### 需求 3.1.1：页面加载性能
- 站点列表页面加载时间应不超过 2 秒
- 充电模拟器页面加载时间应不超过 1 秒
- 数据库查询响应时间应不超过 500 毫秒

#### 需求 3.1.2：实时数据性能
- WebSocket 消息推送延迟应不超过 100 毫秒
- MQTT 消息处理延迟应不超过 50 毫秒
- 系统应支持至少 50 个并发充电会话

#### 需求 3.1.3：缓存性能
- 站点数据应使用本地缓存（Caffeine）
- 充电会话状态应使用 Redis 缓存
- 缓存命中率应不低于 80%

### 3.2 可靠性需求

#### 需求 3.2.1：数据可靠性
- 充电会话数据应持久化到数据库
- MQTT 消息应使用 QoS 1（至少一次）
- 系统应支持异常情况下的状态恢复

#### 需求 3.2.2：连接可靠性
- WebSocket 应支持断线自动重连
- MQTT 连接应支持断线自动重连
- 连接异常应有明确的错误提示

#### 需求 3.2.3：数据一致性
- 充电枪状态应与充电会话状态保持一致
- 实时数据应与 MQTT 消息保持一致
- 数据库事务应保证数据一致性

### 3.3 可用性需求

#### 需求 3.3.1：界面友好性
- 界面应采用左右分栏布局
- 按钮状态应根据充电流程自动启用/禁用
- 未充电时车辆面板应置灰显示
- 数据显示应清晰易读

#### 需求 3.3.2：操作便捷性
- 充电流程应符合实际操作逻辑
- 页面导航应简单直观
- 错误提示应清晰明确

#### 需求 3.3.3：响应式设计
- 页面应支持常见分辨率（1920x1080, 1366x768）
- 数据更新应实时响应
- 按钮点击应有即时反馈

### 3.4 安全性需求

#### 需求 3.4.1：接口安全
- API 接口应进行参数校验
- 应防止 SQL 注入攻击
- 应防止 XSS 攻击

#### 需求 3.4.2：连接安全
- MQTT 连接应进行身份认证
- WebSocket 连接应进行会话验证
- 敏感配置应加密存储

#### 需求 3.4.3：数据安全
- 充电会话数据应有访问控制
- 数据库连接应使用连接池
- 应记录关键操作日志

### 3.5 可维护性需求

#### 需求 3.5.1：代码质量
- 代码应遵循分层架构（Controller-Service-Mapper）
- 应使用统一的异常处理机制
- 应使用统一的响应格式

#### 需求 3.5.2：日志记录
- 应记录关键业务操作日志
- 应记录异常错误日志
- 日志应包含时间戳和上下文信息

#### 需求 3.5.3：配置管理
- 配置应集中管理（application.yml）
- 应支持不同环境的配置切换
- 敏感配置应外部化

## 4. 数据需求

### 4.1 数据实体

#### 实体 4.1.1：充电站（Station）
- 站点ID、站点名称、站点地址、状态
- 与充电桩为一对多关系

#### 实体 4.1.2：充电桩（ChargingPile）
- 充电桩ID、所属站点ID、充电桩编号、状态
- 与充电站为多对一关系
- 与充电枪为一对多关系

#### 实体 4.1.3：充电枪（ChargingGun）
- 充电枪ID、所属充电桩ID、充电枪编号、类型、状态
- 与充电桩为多对一关系

#### 实体 4.1.4：充电会话（ChargingSession）
- 会话ID、充电枪ID、车辆ID、开始时间、结束时间、状态、总电量
- 记录完整的充电过程

### 4.2 数据来源

#### 数据源 4.2.1：数据库
- 充电站、充电桩、充电枪的基础数据
- 充电会话的历史数据
- 数据已存在，需实现查询功能

#### 数据源 4.2.2：MQTT 消息
- 充电实时数据（电压、电流、功率、SOC）
- 车辆 BMS 数据（电池电压、电池电流、电池温度、SOC）
- 数据通过订阅 MQTT 主题获取

### 4.3 数据流转

#### 流程 4.3.1：站点数据流转
```
数据库 → Mapper → Service → Controller → 前端页面
```

#### 流程 4.3.2：实时数据流转
```
MQTT Broker → MqttService → WebSocketService → WebSocket → 前端页面
```

#### 流程 4.3.3：充电会话数据流转
```
前端操作 → Controller → Service → Mapper → 数据库
```

## 5. 接口需求

### 5.1 REST API 接口

#### 接口 5.1.1：获取站点列表
- 方法：GET /api/stations
- 响应：站点列表（包含站点ID、名称、地址、状态）

#### 接口 5.1.2：获取站点详情
- 方法：GET /api/stations/{stationId}
- 响应：站点详情（包含充电桩列表）

#### 接口 5.1.3：获取充电桩详情
- 方法：GET /api/piles/{pileId}
- 响应：充电桩详情（包含充电枪列表）

#### 接口 5.1.4：插枪
- 方法：POST /api/charging/insert-gun
- 请求：充电枪ID、车辆ID
- 响应：充电会话ID

#### 接口 5.1.5：启动充电
- 方法：POST /api/charging/start
- 请求：充电会话ID
- 响应：操作结果

#### 接口 5.1.6：结束充电
- 方法：POST /api/charging/stop
- 请求：充电会话ID
- 响应：充电结果（总电量、充电时长）

#### 接口 5.1.7：拔枪
- 方法：POST /api/charging/remove-gun
- 请求：充电会话ID
- 响应：操作结果

### 5.2 WebSocket 接口

#### 接口 5.2.1：充电数据推送
- 端点：ws://localhost:8080/ws/charging/{sessionId}
- 消息类型：CHARGING_DATA
- 消息内容：电压、电流、功率、SOC、时间戳

#### 接口 5.2.2：BMS 数据推送
- 端点：ws://localhost:8080/ws/charging/{sessionId}
- 消息类型：BMS_DATA
- 消息内容：电池电压、电池电流、电池温度、SOC、时间戳

### 5.3 MQTT 主题

#### 主题 5.3.1：充电数据主题
- 主题：charging/station/{stationId}/pile/{pileId}/gun/{gunId}/data
- 用途：接收充电实时数据

#### 主题 5.3.2：BMS 数据主题
- 主题：vehicle/{vehicleId}/bms/data
- 用途：接收车辆 BMS 数据

## 6. 约束条件

### 6.1 技术约束
- 必须使用 Spring Boot 3.5.11 框架
- 必须使用 MySQL 数据库
- 必须使用 MQTT 协议接收实时数据
- 必须使用 WebSocket 推送数据到前端
- 前后端不分离架构

### 6.2 业务约束
- 一个充电枪同时只能服务一个充电会话
- 充电流程必须按照：插枪 → 启动充电 → 结束充电 → 拔枪 的顺序执行
- 充电数据必须实时更新

### 6.3 环境约束
- 开发环境：Windows/Linux/Mac
- 运行环境：JDK 21
- 数据库：MySQL 8.0+
- MQTT Broker：需外部部署

## 8. 正确性属性（Property-Based Testing）

### 属性 8.1：充电枪状态一致性
**属性描述：** 充电枪的状态应始终与充电会话的状态保持一致

**形式化定义：**
```
∀ gun, session:
  if session.status == CHARGING and session.gunId == gun.id
  then gun.status == CHARGING
```

**测试策略：**
- 生成随机的充电操作序列（插枪、启动、结束、拔枪）
- 验证每次操作后充电枪状态与会话状态的一致性
- 测试并发场景下的状态一致性

### 属性 8.2：充电流程状态转换合法性
**属性描述：** 充电流程的状态转换必须遵循预定义的状态机

**形式化定义：**
```
状态转换规则：
  IDLE --插枪--> INSERTED --启动--> CHARGING --结束--> FINISHED --拔枪--> IDLE
  
非法转换：
  IDLE --启动--> X
  IDLE --结束--> X
  IDLE --拔枪--> X
  INSERTED --结束--> X
  INSERTED --拔枪--> X
  CHARGING --插枪--> X
  CHARGING --拔枪--> X
```

**测试策略：**
- 生成所有可能的状态转换序列
- 验证合法转换能够成功执行
- 验证非法转换被正确拒绝

### 属性 8.3：充电会话唯一性
**属性描述：** 一个充电枪在同一时间只能有一个活跃的充电会话

**形式化定义：**
```
∀ gun, t:
  count(session where session.gunId == gun.id 
        and session.status in [INSERTED, CHARGING] 
        and session.time == t) <= 1
```

**测试策略：**
- 尝试对同一充电枪创建多个并发会话
- 验证系统拒绝重复创建
- 验证会话结束后才能创建新会话

### 属性 8.4：实时数据时序性
**属性描述：** 实时数据的时间戳应单调递增

**形式化定义：**
```
∀ data1, data2 in chargingDataStream:
  if data1 arrives before data2
  then data1.timestamp <= data2.timestamp
```

**测试策略：**
- 生成随机的 MQTT 消息序列
- 验证 WebSocket 推送的数据时间戳顺序
- 测试乱序消息的处理

### 属性 8.5：充电总电量非负性
**属性描述：** 充电会话的总电量必须为非负数

**形式化定义：**
```
∀ session:
  session.totalPower >= 0
```

**测试策略：**
- 生成各种充电时长和功率的测试用例
- 验证计算出的总电量始终非负
- 测试边界情况（0 时长、0 功率）

### 属性 8.6：WebSocket 连接与会话绑定
**属性描述：** WebSocket 连接应正确绑定到充电会话，数据推送应发送到正确的客户端

**形式化定义：**
```
∀ wsConnection, session, data:
  if wsConnection.sessionId == session.id
  then data pushed to wsConnection should belong to session
```

**测试策略：**
- 创建多个并发充电会话和 WebSocket 连接
- 验证每个连接只接收到对应会话的数据
- 测试连接断开和重连场景
